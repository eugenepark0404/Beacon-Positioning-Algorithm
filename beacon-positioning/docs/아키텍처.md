# 아키텍처 — 데이터 흐름과 트랙 간 인터페이스

## 1. 전체 데이터 흐름

```
[수신부]                [코어 파이프라인 (ble_positioning/)]                       [소비자]
Android(Kotlin) ─┐
iOS(Swift)      ─┼─► ingest ─► preprocessing ─► correction ─► distance ─► positioning ─► output ─► 트랙5 UX
Unity 트랙4     ─┘   (로더)    (창집계·median·   (차폐 감지·    (RSSI→거리)  (NLS 다변측량    (JSON/CSV)
                              칼만·MAD)         보정·다중경로)              +잔차 보정,
                                                                          WCL/그리드/KNN)
                                                                              │
                                                    트랙2 센서퓨전 ◄── fusion (PF 골격 + PDR 인터페이스)
```

처리 순서 (pipeline.py `process()`):
1. `group_scan_windows` — scan_window_s(기본 1s) 창으로 시계열 분할
2. (옵션, `use_channel_rule=True`) 채널 패턴 차폐 감지·보정 — P1 원리, 채널 정보 필요
3. `aggregate_by_beacon` — MAD 이상치 제거 + 비콘별 중앙값 (P3)
4. `RssiKalman1D` — 스칼라 칼만 스무딩 (P4)
5. `LogDistanceModel` — RSSI→거리 (P1 Eq.1)
6. 위치 추정 — 기본 NLS 다변측량 (P1), 실패 시 WCL fallback (P3)
7. **잔차 기반 차폐/NLoS 2차 보정** — 거리 과대추정 비콘 재보상 후 재적합 (기본 ON)
8. `PositionRecord` 출력 — CSV/JSON

## 2. 데이터 포맷 (전 트랙 합의 — 변경 시 주 1회 인터페이스 동기화 회의에서)

### 입력 (트랙4 Unity / 실측 공용) — `track4-unity-v1`

CSV:
```
timestamp,beacon_id,rssi,channel,x_true,y_true,noise_flag
12.400,B03,-71.25,38,3.512,2.000,shadow
```
- `timestamp` [s] float — 시뮬 상대시간 또는 epoch
- `beacon_id` str — 비콘 좌표 맵의 키와 일치
- `rssi` [dBm] float
- `channel` 37|38|39|빈칸 — 폰 실측은 빈칸(플랫폼이 미노출), 시뮬은 채움
- `x_true,y_true` [m] — 정답 좌표 (시뮬·평가 전용, 실측은 빈칸)
- `noise_flag` none|shadow|multipath — 트랙4 주입 오차 라벨 (평가 전용)

JSON: `{"meta": {"format": "track4-unity-v1", ...}, "samples": [{"t","beacon_id","rssi","channel",...}]}`

### 출력 (트랙2·5 소비)

CSV: `timestamp,x,y,method,n_beacons,shadow_corrected,residual,x_true,y_true`

### 비콘 좌표 맵

`beacon_map.json`: `{"B01": [x, y], ...}` — 단위 m. **좌표계 원점·축 방향은 트랙4
Unity 씬과 트랙5 3D 모델이 공유하는 기준을 따른다** (합의 문서에 고정할 것).

## 3. 트랙별 연동 지점

| 트랙 | 인터페이스 | 파일 |
|------|-----------|------|
| 트랙2 (센서퓨전) | `FusionEngine` ABC: `update_ble(BlePositionInput)` / `update_pdr(PdrStep)` / `estimate()`. PF 골격(`ParticleFilter2D`, P2 수식)과 CV-칼만 골격 제공. `map_constraint` 훅 = P5 도면 사후보정 지점 | `fusion/interfaces.py`, `fusion/particle_filter.py` |
| 트랙3 (하드웨어·실측) | 캘리브레이션 상수 주입: `config/*.json` → `load_config()`. 절차는 캘리브레이션_가이드.md | `config/settings.py` |
| 트랙4 (Unity 시뮬) | 입력 포맷 `track4-unity-v1` 준수. 합성 생성기(`examples/generate_synthetic.py`)가 포맷 레퍼런스 구현 | `ingest/loaders.py` |
| 트랙5 (UX) | 출력 CSV/JSON 구독. `residual` 필드 = 신뢰도 지표(경고 UX에 활용 가능) | `output/writers.py` |

## 4. 수신부 ↔ 코어 경계

- 수신부(Android/iOS)는 **원시 RSSI 샘플 수집·포맷 변환만** 담당. 필터·보정·측위는 코어.
- 온디바이스 측위(기본 방침): 코어 알고리즘의 Kotlin/Swift 이식은
  `android/.../PositioningBridge.kt` 인터페이스 뒤에서 진행 (이식 순서 명시됨).
- 플랫폼 제약: 폰은 광고 수신 채널 번호를 노출하지 않음 → P1 3채널 차폐 감지는
  시뮬·전용 수신기 검증용, 폰 실전은 잔차 보정 + (학습 후) ANN 보정.
